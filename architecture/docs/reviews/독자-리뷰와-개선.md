# 독자 리뷰와 개선 기록

본문 15장 + 서문 + 부록 초고 완성 후, 두 명의 서로 다른 연차 독자 페르소나에게 책 전체를 읽히고 받은 리뷰, 그리고 그에 대한 개선을 기록한다. (사실 검증 감수와는 별개의 "독자 효용" 관점 리뷰다.)

---

## 1. 두 리뷰어의 판정 — 둘 다 ★★★★☆ (4.0/5)

흥미롭게도 5년차와 15년차 모두 4.0을 줬지만, **별을 깎은 이유가 서로 달랐다.**

| | 5년차 (대상 독자) | 15년차 (시니어/추천자) |
|---|---|---|
| 총평 | "패턴의 '왜'를 이보다 정직하게 설명한 책은 드물다. 다만 통찰의 절반이 코드로 착지 못한 채 끝난다" | "'유행이 아니라 고통에서 출발하라'를 15장 내내 배신 안 한 서사서 — 다만 'Deep Dive'는 분산·데이터·운영 앞에서 절반만 참" |
| 감점 이유 | 코드가 얇고·늦고·장난감 / Step 10~11 개념 폭격에 익사 | 무거운 주제가 제목만큼 안 깊음 / 운영 현실 경시 / 중심 실증 미이행 |
| 추천 | 미들에게 강력 추천, 스터디 교재 추천 | 미들에게 Yes, 스터디 Yes(단 분산은 보강), 시니어엔 "새 지식보다 팀 공유 어휘" |

## 2. 공통으로 칭찬한 것 (= 확실한 강점)

두 사람이 **독립적으로** 같은 것을 꼽았다:

1. **"화살표"(Step 03)가 전권을 꿰뚫는 등뼈.** 특히 DIP를 "추상 소유권 역전"으로 정정한 것을 둘 다 "시중 책이 놓치는 지점"이라 평가.
2. **"고통 → 응답 → 새 고통" + 수렴/진자 프레임이 실제로 작동.** 카탈로그가 아닌 서사. 15년차: "Ford/Richards조차 이보다 카탈로그에 가깝다."
3. **하이프를 안 파는 정직성.** "작을 땐 레이어드가 이긴다", 시나리오 B 자기 격하, ArchUnit 실패담.
4. **오해 교정 절이 실전 진단 도구.**
5. (15년차 고유) **사료 정확성** — 부록 C 연표, NATO 1968 통설 반박, Yegge 전거.

## 3. 공통으로 지적한 약점 (= 우연이 아닌 신호)

두 사람이 똑같이 짚은 두 가지:

1. **Step 10~11(분산·데이터)의 깊이 부족.** 5년차 "개념 폭격에 익사", 15년차 "breadth-first 개론, Deep Dive엔 못 미침". 같은 현상의 양면 — 초심자엔 너무 빠르고 시니어엔 너무 얕다.
2. **Part 3 코드가 중심 실증을 이행 못 함.** 5년차 "가장 값진 Step 11이 코드 0줄", 15년차 "diff로 입증되는 건 시나리오 A뿐, 정작 중심 주장인 시나리오 C가 코드로 없다".

## 4. 시각이 갈린 지점 (연차 차이)

- **Step 09~10(조직/MSA)**: 5년차는 "팀 재편 권한이 없어 관객이 된다"(실무 거리감), 15년차는 "'MSA의 진짜 계기는 조직'은 방어 가능한 테제이지 합의된 사실이 아닌데 정설처럼 처리"(비평적 거리감).
- **15년차 고유 지적**: 운영 현실 경시(관측가능성·배포·비용·on-call이 한 장도 못 받음), 보안 실종, 비용이 다이얼에서 빠짐, 서사가 헥사고날/DDD로 은근히 기움("레이어드는 졸업할 출발점으로 프레이밍").
- **5년차 고유 지적**: 예제가 전부 Java(코틀린 마찰), 다이어그램 빈약, 산문 밀도가 살인적("재독에서 빛나지 초독엔 불친절").

## 5. 개선 반영

두 리뷰가 독립적으로 수렴한 신호(3절)를 우선순위로 삼아 반영했다.

### 반영 완료

- **Step 11 코드 보강** (커밋 `6bd5c7f`). `data` 패키지에 트랜잭셔널 아웃박스·멱등 수신·추가 전용 원장·역분개를 실제 코드로 구현하고, 이중 쓰기 문제를 대조하는 테스트(`DualWriteTest`)와 멱등성(`IdempotencyTest`)·원장(`LedgerTest`)로 증명. "코드 0줄"이라던 최대 불만을 정면 해소. step11 문서에 코드 절 추가.
- **다이어그램** (커밋 `6bd5c7f`). 부록 E를 신설해 Mermaid로 레이어드↔헥사고날 의존 그래프, 어니언/클린 동심원, 이중쓰기/아웃박스 시퀀스, 진자→나선, 삼중 정렬을 시각화. GitHub에서 렌더링됨. 핵심 대비 그림을 Step 13에 직접 임베드. "시각자료 빈약" 지적 해소.
- **시나리오 C 코드 시연** (커밋 `d0e8302`). 15년차가 "중심 주장인데 코드로 없는 유일한 것"이라 지적한 규칙 축적→페이오프 역전을 `growth` 패키지로 구현. 레이어드는 규칙 드리프트로 고액 주문이 무승인 취소되는 버그(`LayeredDriftTest`), 헥사고날은 단일 장소로 그 버그가 불가능함(`HexagonalSinglePlaceTest`)을 대조. 스트로맨 방지를 위한 "정직한 각주"(규율 있는 팀은 회피 가능) 포함. 감수 결과 "공정한 시연 — 스트로맨 아님, 비대칭은 빈혈 엔티티의 열린 setter라는 진짜 구조 차이에서 나옴" 판정, LOW 2건(각주에 "캡슐화가 유일한 구조적 수리" 명시 + 변경 amplification 상수-대-선형 기제 한 줄)을 반영해 논증을 airtight로.
- **정직성 주석**: 데이터 코드에 릴레이 동시성·멱등 집합 무한 증가라는 단순화의 한계를 `ponytail:` 주석으로 명시(책의 정직성 습관과 정합).

### 의도적으로 남긴 것 (스코프 결정)

- **분산의 깊이(합의·격리 수준·샤딩)**: 이 책은 "왜"의 책이지 분산 시스템 교과서가 아니다. 부록 D에서 Kleppmann으로 명시적으로 넘긴다. Deep Dive라는 제목과의 긴장은 인정하되, 한 권이 모든 깊이를 담을 수 없다는 판단.
- **운영/보안/비용을 독립 장으로**: 15년차의 정당한 지적이나, 추가하면 책의 초점(구조와 화살표)이 흐려진다. 향후 개정에서 Step 04(품질 속성)나 Step 15(다이얼)에 보안·비용을 항목으로 승격하는 선에서 검토.
- **코틀린 예제**: 프로젝트가 코틀린을 쓰지만, 교재 예제는 범용 가독성을 위해 Java 유지. 논지는 언어 중립임을 서문에 명시.
